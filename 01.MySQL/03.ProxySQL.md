## ProxySQL:

ProxySQL is an open-source, high-performance MySQL proxy that sits between your application and your MySQL/MariaDB database servers.


### What it actually does:

- **Connection pooling** — maintains a pool of reused connections to MySQL, drastically reducing the overhead of thousands of apps opening/closing connections
- **Read/write splitting** — automatically routes SELECT queries to replicas and writes to the primary based on rules you define
- **Query routing** — send specific queries to specific servers based on regex pattern matching
- **Query caching** — cache query results in memory to reduce backend load
- **Query rewriting** — modify queries on the fly before they hit MySQL, without touching application code
- **Failover handling** — monitors backend health and reroutes traffic away from failed nodes automatically
- **Query firewall** — block or flag dangerous or unexpected queries
- **Statistics and observability** — detailed per-query stats, connection pool metrics, and a query digest showing what's hitting your database



### Architecture overview:

```
               ┌──────────────┐
Application →  │  ProxySQL     │
               └──────┬───────┘
                      │
        ┌─────────────┴─────────────┐
        │                           │
   Primary DB                 Replica DBs
(write)                     (read scaling)
```


### Prerequisites:
- Supported OS — Linux: Ubuntu, Debian, CentOS, or RHEL
- MySQL / MariaDB — Must be installed and running before configuring ProxySQL backends
- Allow ports:
    - Port 6033 — application traffic (MySQL protocol)
    - Port 6032 — admin interface (MySQL protocol)
    - Port 6080 — optional web UI

## Install ProxySQL:


### Official ProxySQL repo:


```
sudo tee /etc/yum.repos.d/proxysql.repo <<EOF
[proxysql]
name=ProxySQL repo
baseurl=https://repo.proxysql.com/ProxySQL/proxysql-2.6.x/centos/8/
gpgcheck=1
gpgkey=https://repo.proxysql.com/ProxySQL/proxysql-2.6.x/repo_pub_key
enabled=1
EOF
```




```
yum install -y proxysql
```


```

```

### Add MySQL backend servers:

- Hostgroup 10 = primary (writes)
- Hostgroup 20 = replicas (reads)


_Add MySQL servers to the `mysql_servers` table, specifying a `hostgroup`, `hostname`, and `port` :_
```
--> Primary
INSERT INTO mysql_servers (hostgroup_id, hostname, port) VALUES (10, '192.168.10.191', 3306);  

--> Replica
INSERT INTO mysql_servers (hostgroup_id, hostname, port) VALUES (20, '192.168.10.192', 3306);  
INSERT INTO mysql_servers (hostgroup_id, hostname, port) VALUES (20, '192.168.10.193', 3306); 
```



_Set replica weights for load balancing:_
```
update mysql_servers set weight = 100 where hostname='192.168.10.192';

update mysql_servers set weight = 100 where hostname='192.168.10.193';
```


_Apply changes:_
```
LOAD MYSQL SERVERS TO RUNTIME;
SAVE MYSQL SERVERS TO DISK;
```

### Add MySQL Users:
- `default_hostgroup=10` - means writes go to the primary. 
- `transaction_persistent=1` - keeps a transaction on the same backend node for consistency.

_Create application users in **MySQL**:_
```
CREATE USER 'appuser'@'%' IDENTIFIED BY 'Secre#t123';
GRANT ALL PRIVILEGES ON *.* TO 'appuser'@'%';
```


_Add them to **ProxySQL’s** `mysql_users` table:_
```
INSERT INTO mysql_users (username, password, default_hostgroup, transaction_persistent)
VALUES ('appuser', 'Secre#t123', 10, 1);
```


_Apply changes:_
```
LOAD MYSQL USERS TO RUNTIME;
SAVE MYSQL USERS TO DISK;
```



```

```





### Configure query routing rules (Read/Write Splitting): 

- SELECT FOR UPDATE stays on primary
- Route SELECT queries to the read hostgroup (20)
- All other SELECTs go to replicas


_Writes → Primary (hostgroup 10):_
```
### Special SELECT → go to PRIMARY (Locking SELECT → Primary):
INSERT INTO mysql_query_rules (rule_id, active, match_pattern, destination_hostgroup, apply) VALUES (1, 1, '^SELECT.*FOR UPDATE', 10, 1);


INSERT INTO mysql_query_rules (rule_id, active, match_digest, destination_hostgroup, apply) VALUES (2, 1, '^(INSERT|UPDATE|DELETE|REPLACE)', 10, 1);
```



_Read → Replica (hostgroup 20):_
```
INSERT INTO mysql_query_rules (rule_id, active, match_digest, destination_hostgroup, apply) VALUES (3, 1, '^SELECT', 20, 1);
```



_Handle transactions, without this, you can break consistency:_
```
INSERT INTO mysql_query_rules (rule_id, active, match_digest, destination_hostgroup, apply) VALUES (4, 1, '^(BEGIN|COMMIT|ROLLBACK)', 10, 1);
```



_Apply changes:_
```
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL QUERY RULES TO DISK;
```



```
select rule_id, active, match_digest, destination_hostgroup, apply from mysql_query_rules;

+---------+--------+---------------------------------+-----------------------+-------+
| rule_id | active | match_digest                    | destination_hostgroup | apply |
+---------+--------+---------------------------------+-----------------------+-------+
| 1       | 1      | NULL                            | 10                    | 1     |
| 2       | 1      | ^(INSERT|UPDATE|DELETE|REPLACE) | 10                    | 1     |
| 3       | 1      | ^SELECT                         | 20                    | 1     |
+---------+--------+---------------------------------+-----------------------+-------+
```


```
select rule_id, active, match_pattern, destination_hostgroup, apply from mysql_query_rules;

+---------+--------+---------------------+-----------------------+-------+
| rule_id | active | match_pattern       | destination_hostgroup | apply |
+---------+--------+---------------------+-----------------------+-------+
| 1       | 1      | ^SELECT.*FOR UPDATE | 10                    | 1     |
| 2       | 1      | NULL                | 10                    | 1     |
| 3       | 1      | NULL                | 20                    | 1     |
+---------+--------+---------------------+-----------------------+-------+
```


#### Delete rules:

```
DELETE FROM mysql_query_rules WHERE rule_id IN (1);
DELETE FROM mysql_query_rules WHERE rule_id IN (2);
```



#### What happens if you don’t fix?

- select ... for update might go to replica ❌
- lowercase queries won’t match ❌
- transactions may split across nodes ❌
- inconsistent data ❌


```
--> Locking reads → PRIMARY
(1, 1, '(?i)^SELECT.*FOR UPDATE', 10, 1)

--> Writes → PRIMARY
(2, 1, '(?i)^(INSERT|UPDATE|DELETE|REPLACE)', 10, 1)

--> Transactions → PRIMARY
(3, 1, '(?i)^(BEGIN|COMMIT|ROLLBACK)', 10, 1)

--> Normal reads → REPLICA
(4, 1, '(?i)^SELECT', 20, 1)
```
